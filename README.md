## What is AURA Forge?

AURA Forge is an AI-powered smart contract development platform that takes you from a plain-English description to a compiled, security-audited, and deployed smart contract — without leaving the browser.

It supports two ecosystems:

- **EVM** — Real Solidity source compiled with `solc`, with a self-healing LLM retry loop that automatically fixes compiler errors and feeds them back into the model. Deployable to Ethereum, Base, Arbitrum, Optimism, and Polygon.
- **Solana** — Real Anchor/Rust source with a full IDL, generated by Claude and ready to be dropped into a local Anchor workspace for on-chain deployment.

The security hardening engine runs a dedicated auditor LLM pass over every generated contract, scores it 0–100, and iterates until it crosses a configurable threshold — or asks you for more business context when it needs it to go further.

## Feature Overview

| Feature | Description |
|---|---|
| **AI Generation** | Prompt → Solidity or Anchor/Rust source code via Claude |
| **Real Compilation** | `solc` compiles EVM contracts; errors feed back into the LLM (up to 3 self-healing retries) |
| **Security Hardening** | Dedicated auditor LLM pass, 0–100 score, iterative improvement |
| **Live Progress Console** | SSE-streamed build log in the UI — every phase visible in real time |
| **One-Click Deployment** | Browser wallet (MetaMask/viem) for EVM testnets and mainnet |
| **Solana Devnet Broadcast** | Phantom wallet verification broadcast for Solana devnet |
| **Code Editor** | Monaco-based editor — fully editable post-generation; edits persist to the project and invalidate stale analysis |
| **Gas Analysis** | Solc-grounded gas estimates per function (EVM); LLM-authored notes for Solana |
| **Source Verification** | Automatic Etherscan/Blockscout source verification after EVM deploy |
| **Post-Deploy Monitoring** | Webhook and email alerts for deployed contract activity |
| **Version History** | Full lineage of every hardening pass with inline diff viewer |
| **Team Workspaces** | Multi-user teams, invite flows, role-based project access |
| **API Keys** | Programmatic access for CI/CD integration |
| **Templates** | Starter templates: ERC-20, NFT, DAO, Vesting, Multisig, and more |
| **Project Export** | Download the full project as a ZIP for local Foundry/Anchor development |
| **Upgradeable Contracts** | Optional UUPS proxy pattern for EVM contracts |

## Architecture

```
workspace/
├── artifacts/
│   ├── aura-forge/          # React + Vite SPA (Monaco editor, wallet connect, dashboard)
│   └── api-server/          # Express 5 REST API (forge pipeline, auth, teams, monitoring)
│
└── lib/
    ├── api-spec/            # OpenAPI spec (openapi.yaml) + Orval codegen config
    ├── api-client-react/    # Generated React Query hooks (auto-generated from spec)
    ├── api-zod/             # Generated Zod schemas (auto-generated from spec)
    ├── db/                  # Drizzle ORM schema + migration config
    └── integrations-anthropic-ai/  # Anthropic Claude wrapper (Replit AI Integrations proxy)
```

### Forge Pipeline (`api-server/src/lib/forge/`)

```
User prompt
    │
    ▼
LLM generates source (Claude)
    │
    ├─[EVM]──► solc compile ──► errors? ──► LLM self-heal (≤3 retries) ──► success
    │
    └─[Solana]──► IDL extraction ──► validation
    │
    ▼
Auditor LLM ──► security score (0-100) ──► below target?
    │                                            │
    │                                     ◄──── LLM hardens
    │
    ▼
Gas estimates + notes ──► ready to deploy
```

The hardening loop can pause and surface a targeted context question to the user when the auditor determines that additional business logic information would raise the score further.

## Tech Stack

### Frontend (`artifacts/aura-forge`)
- **React 19** + **Vite** — SPA with HMR
- **Wouter** — lightweight client-side routing
- **TanStack Query** — server state, caching, and mutations

### Backend (`artifacts/api-server`)
- **Express 5** — REST API
- **PostgreSQL** + **Drizzle ORM** — type-safe database access
- **connect-pg-simple** — PostgreSQL-backed sessions
- **bcryptjs** — password hashing
- **solc** — real EVM compilation

- **Pino** — structured logging

- **OpenAPI + Orval** — API-first development; all endpoints defined in `lib/api-spec/openapi.yaml`, REST hooks and Zod schemas auto-generated via `pnpm codegen`
- **Anthropic Claude** — via Replit AI Integrations proxy (no direct key management needed in production)

## Getting Started

### Prerequisites

- **Node.js** ≥ 20
- **pnpm** ≥ 9
- **PostgreSQL** database
- **Anthropic API key** (or access via Replit AI Integrations)

### 1. Clone the repository

```bash
git clone https://github.com/Harmain11/Cross-Chain-Hub.git
cd Cross-Chain-Hub
```

### 2. Install dependencies

```bash
pnpm install
```

### 3. Set environment variables

Create a `.env` file in the root (or set these in your host environment):

```env
# PostgreSQL connection string
DATABASE_URL=postgresql://user:password@localhost:5432/aura_forge

# Session secret — any long random string
SESSION_SECRET=your-secret-here-min-32-chars

# Anthropic API access (via Replit AI Integrations or direct)
AI_INTEGRATIONS_ANTHROPIC_API_KEY=your-anthropic-key
AI_INTEGRATIONS_ANTHROPIC_BASE_URL=https://api.anthropic.com
```

### 4. Push the database schema

```bash
pnpm --filter @workspace/db run push
```

> **Note:** `drizzle-kit push` is for development only. For production, generate and apply migrations instead.

### 5. Start the development servers

In two separate terminals:

```bash
# Terminal 1 — API server (port 8080 by default, reads $PORT)
pnpm --filter @workspace/api-server run dev

# Terminal 2 — Frontend (reads $PORT)
pnpm --filter @workspace/aura-forge run dev
```

## Development Workflows

| Command | Description |
|---|---|
| `pnpm --filter @workspace/api-server run dev` | Start API server in development mode |
| `pnpm --filter @workspace/aura-forge run dev` | Start frontend Vite dev server |

| `pnpm run build` | Typecheck + build all packages |
| `pnpm --filter @workspace/api-spec run codegen` | Regenerate React Query hooks and Zod schemas from `openapi.yaml` |
| `pnpm --filter @workspace/db run push` | Push DB schema changes (development only) |

### Modifying the API

All REST endpoints are contract-first: edit `lib/api-spec/openapi.yaml`, then run codegen to update the client hooks and server-side Zod validation schemas automatically:

```bash
pnpm --filter @workspace/api-spec run codegen
```

This regenerates:
- `lib/api-client-react/src/generated/` — React Query hooks used by the frontend
- `lib/api-zod/src/generated/` — Zod schemas used for request/response validation in the API server

## Key Design Decisions

### EVM Compilation is Real
Claude generates Solidity source, the `solc` npm package compiles it, and any compiler errors are fed verbatim back to the LLM for self-healing (up to 3 attempts). There is no mock or simulation path for EVM — if it doesn't compile, it doesn't pass.

### Solana Compilation is Simulated (and Labeled As Such)
The platform generates production-quality Anchor/Rust source and a full IDL. However, there is no `cargo`/Anchor toolchain available at runtime, so no `.so` binary is built. The generated source is ready for immediate local compilation via `anchor build`. Solana "deployment" is a real devnet transaction broadcast from the user's Phantom wallet, explicitly labeled as a verification broadcast, not a full program deploy.

### No Constructor Arguments
Generated contracts are constrained via the LLM system prompt to have no required constructor arguments. This keeps the deployment UI simple — no dynamic argument form is ever needed.

### Code Editing Invalidates Analysis
When a user manually edits a generated contract through the Monaco editor and saves, the stored security score, security notes, gas estimates, compiled bytecode, compile log, and verification status are all cleared server-side. This prevents stale analysis from being shown alongside hand-edited code that was never re-audited. Re-run "Improve Security" or redeploy to refresh.

### API-First Development
The entire REST surface lives in one `openapi.yaml` file. Frontend hooks and server validation schemas are generated from it — no drift between client and server types.

### Authentication
Custom email/password authentication using `bcryptjs` and PostgreSQL-backed sessions (`express-session` + `connect-pg-simple`). No third-party auth provider dependency.

## API Reference

The full API is described in [`lib/api-spec/openapi.yaml`](lib/api-spec/openapi.yaml). Key endpoint groups:

| Group | Endpoints |
|---|---|
| **Auth** | `POST /api/auth/signup`, `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/auth/me` |
| **Projects** | `GET /api/projects`, `GET /api/projects/:id`, `DELETE /api/projects/:id` |
| **Forge** | `POST /api/forge-contract`, `GET /api/forge-contract/:id/stream` (SSE) |
| **Harden** | `POST /api/projects/:id/harden` |
| **Code Edit** | `PATCH /api/projects/:id/code` |
| **Deploy** | `PATCH /api/projects/:id/deploy` |
| **Monitoring** | `PATCH /api/projects/:id/monitoring` |
| **Lineage** | `GET /api/projects/:id/lineage` |
| **Export** | `GET /api/projects/:id/export` |
| **Teams** | `GET /api/teams`, `POST /api/teams`, `GET /api/teams/:id/members` |
| **Invites** | `POST /api/teams/:id/invite`, `GET /api/invites`, `POST /api/invites/:id/accept` |
| **API Keys** | `GET /api/api-keys`, `POST /api/api-keys`, `DELETE /api/api-keys/:id` |

The forge streaming endpoint (`/api/forge-contract/:id/stream`) uses Server-Sent Events and is intentionally excluded from the OpenAPI spec and codegen pipeline, since SSE cannot be expressed cleanly in the React Query hook pattern.
